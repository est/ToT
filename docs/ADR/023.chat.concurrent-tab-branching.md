## Context

用户可能在多个 Tab 打开同一个会话。如果 Tab A 和 Tab B 几乎同时发送消息，会发生什么？

当前实现中，两个 Tab 都会：
1. 查询 latest node
2. 用它作为 parent
3. 插入新节点

这导致两个消息成为 sibling 节点（同一 parent 的分支），而非顺序关系。

候选方案：

- A: 自动分支 — 并发消息变成 sibling 分支，符合树结构设计
- B: 乐观锁 — 检测 latest 变化后重试，强制线性
- C: 防并发 — 检测其他 Tab 编辑时禁用输入

## Decision

采用方案 A：并发消息自动成为分支。

理由：

1. Tree of Threads 的核心设计就是树结构，分支是原生能力
2. 无需复杂锁机制，实现简单
3. 用户可以选择继续哪个分支，符合"思维树"的使用模式
4. 与现有 `/send` 和 `/send/stream` 端点的 prefix_idx 计算逻辑天然兼容

行为描述：

- Tab A 发送消息 → 创建 node X，parent 为 latest
- Tab B 发送消息 → 创建 node Y，parent 也为 latest
- X 和 Y 成为 sibling 分支
- 两个 Tab 各自看到自己创建的分支
- 刷新后可在树视图中看到完整分支结构

## Consequences

优点：

- 实现成本为零，现有代码已支持
- 并发变为分支，不是错误，是特性
- 符合 Tree of Threads 的设计理念

缺点：

- 用户可能预期线性行为，需要理解分支概念
- 两个 Tab 看到的内容可能不一致（直到刷新）
