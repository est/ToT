## Context

需要在 cookie 中存储 session 信息。候选：

- 无状态 JWT token
- userId + random token 的字符串组合
- binary pack → base64

要求：
- 可以从 cookie 直接解析出 userId
- 不泄露信息（不过多暴露结构）
- 紧凑

## Decision

Cookie 格式：binary pack → base64。

```
[4 bytes: userId (uint32 big-endian)] [16 bytes: random token]
→ base64 → Cookie: session=<base64string>
```

解析时：base64 decode → 拆出 userId（前 4 字节）+ token hex（后 16 字节）→ 查 user → 验证 token。

## Consequences

优点：

- 紧凑（24 字节 → 32 字符 base64）
- userId 可直接从 cookie 解析，不需要额外查表
- token 不可预测（16 字节随机）

缺点：

- 非标准格式（vs JWT），需要自定义解析
- 无法在客户端读取（HttpOnly）
- token 没有签名（安全性依赖于 D1 查询验证）
