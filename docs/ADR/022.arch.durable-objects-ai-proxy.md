## Context

当前 AI 调用链路：

```
Browser → Worker (Hono) → fetch(AI API) → Worker → Browser
```

核心问题：

1. **连接脆弱** - Worker 同步等待 AI 响应，浏览器断开则响应丢失
2. **无流式** - 用户等到全部完成才看到结果，体验差
3. **超时限制** - Worker 有执行时间限制，长任务会失败
4. **无编排** - 无法串联/并联多个 AI 调用（子代理）

候选方案：

- A: Worker 直接流式调用（SSE，无 DO）
- B: Durable Objects 作为 AI 代理层
- C: 外部队列服务（如 Cloudflare Queues）

## Decision

采用方案 B：Durable Objects 作为 AI API 代理层。

架构变更：

```
Browser ←SSE← Worker ←→ DO ←streaming→ AI API
          (视图层)    (有状态)
```

核心设计：

1. **DO 键控** - 每个对话（conv_id）绑定一个 DO 实例
2. **SSE 流式** - Worker 返回 SSE，DO 推送 AI chunks
3. **断线恢复** - DO 继续执行，结果存 D1，浏览器重连后获取
4. **子代理** - DO 内部可编排多个 AI 调用

## Consequences

优点：

- 连接韧性：浏览器断开不影响 AI 执行
- 流式体验：SSE 推送 chunks，实时看到输出
- 子代理能力：DO 内部可并行/串联多个 AI 调用
- 无超时限制：DO 没有 Worker 的执行时间限制
- 现有 API 兼容：Worker 层可以保持 REST 接口不变

缺点：

- 复杂度增加：多了一层，调试链路变长
- DO 冷启动：首次调用有 ~10ms 延迟
- 成本变化：DO 计费与 Worker 不同，需要监控
- 数据迁移：现有对话数据需要兼容新架构

依赖：

- ADR-002（Cloudflare Workers 运行时）
- ADR-003（Hono 框架）

替代方案被拒原因：

- 方案 A（纯 Worker SSE）：无法解决断线恢复和子代理需求
- 方案 C（外部队列）：引入外部依赖，增加运维复杂度
