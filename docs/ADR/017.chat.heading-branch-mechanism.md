## Context

ADR-014 确定了"assistant 的 markdown 标题自动成为分支入口"的概念。需要定具体实现方式。

候选：

- 后端解析 markdown，提取 heading 节点，返回结构化数据，前端渲染
- 前端渲染 markdown 后，DOM 查询 h2 元素，绑定事件
- 前端渲染前，正则替换 `## xxx` 为带 onclick 的 HTML

## Decision

采用正则替换方案。

流程：
1. LLM 返回完整 markdown 文本
2. 后端 `parseHeadings()` 用正则 `/^##\s+(.+)/gm` 提取标题列表，随响应返回
3. 前端用 `marked.parse()` 渲染为 HTML
4. 前端对渲染后的 HTML 做正则替换：`<h2>xxx</h2>` → `<h2 class="branch-heading" onclick="branch(id, 'xxx')">▸ xxx</h2>`

不在后端做 HTML 渲染。markdown 解析全在前端。

## Consequences

优点：

- 后端只存原始 markdown，不依赖渲染库
- 前端用 CDN marked.js，零构建依赖
- 正则替换简单直接

缺点：

- 正则替换可能误伤（代码块里的 `##` 会被匹配）— MVP 可接受
- marked.js 版本升级可能改输出格式 — 锁定版本可缓解
- 后续如需更精确控制，可改用 marked 的 renderer API
